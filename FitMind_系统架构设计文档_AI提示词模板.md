# FitMind 系统架构设计文档 AI 提示词模板

## 使用说明

- 先将“通用总提示词”发送给 AI。
- 再根据要扩写的章节，单独发送对应“小节提示词”。
- 如需提高稳定性，可一并附上“参考材料清单”。
- 建议按章节分次生成，不要一次性要求 AI 输出整篇全文。

## 一、通用总提示词

```md
你现在要撰写《FitMind 系统架构设计文档》正文，要求严格基于我提供的项目事实写作，不允许臆造不存在的系统、模块、技术组件或部署方式。

已确认的项目事实如下：
1. 项目名称：FitMind
2. 架构形态：前后端分离 + 后端单体模块化架构，不是微服务
3. 前端项目：fitmind-web
4. 前端技术栈：Vue 3、TypeScript、Vite、Pinia、Vue Router、Element Plus、Axios、Three.js、ECharts
5. 后端项目：fitmind-server
6. 后端技术栈：Java 17、Spring Boot 3.2.3、Spring Security、MyBatis-Plus、JWT、MySQL、Redis
7. 后端主要业务模块：用户、身体档案、AI、训练、饮食、动作识别、食物识别、社区、成就、通知、仪表盘、管理后台
8. AI 对话、训练计划、饮食计划主要通过 DeepSeek API 实现
9. 饮食识别通过 Qwen Vision API 实现
10. 动作识别通过 Python 推理脚本与模型实现，后端通过进程调用 Python
11. 实时动作识别通过 WebSocket + Python 实时推理进程实现
12. 天气能力依赖前端高德天气 API
13. 数据库核心表包括：sys_user、user_body_profile、user_body_metric_log、ai_training_plan、ai_diet_plan、chat_message、community_post、community_comment、community_post_like、user_follow、achievement、user_achievement、exercise、food_item、food_record、notification
14. 后端鉴权方式：Spring Security + JWT，登录/注册开放，其余接口默认需认证，WebSocket 路径 /ws/** 放行后再在握手阶段做鉴权
15. 前端通过 /api 访问后端，通过 /ws 访问实时动作识别

写作要求：
1. 使用正式、学术化、课程设计/毕业设计可直接采用的中文书面语
2. 输出 Markdown
3. 每个小节都写成可直接进入文档的正文，不要写成提纲
4. 先写总体说明，再分点展开，必要时用表格
5. 对不确定的内容，不要编造，统一写成“可作为后续优化方向”或“待进一步部署验证”
6. 不要把当前项目写成微服务、消息队列架构、Kubernetes 架构，除非明确标注为“后续可扩展方案”
7. 结合健身平台业务场景来写，不要只写通用套话
8. 需要体现项目当前真实实现细节，例如：DeepSeek、Qwen、Python 动作识别、WebSocket、JWT、Redis、MySQL
9. 如果涉及插图，请用“图x-x xxx（此处插入图片）”的格式预留图位，并配套写图说明
10. 如果涉及接口、表、模块，请尽量结合 FitMind 的实际命名来写，且表要有表名，图要有图名

请只输出我本次要求的章节正文，不要输出全文。
```
